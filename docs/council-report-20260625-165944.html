LLM Council Report: Async Imperial RAG Evals

Question

What is the right implementation strategy to make Imperial RAG evals genuinely asynchronous while preserving CLI usability, metric behavior, tests, and Phoenix compatibility?

Chairman's Verdict

Implement async where it can be real: the standalone Ragas path. Add async primitives in src/imperial_rag/evals/ragas.py, add evaluate_ragas_rows_async(...) in scripts/run_ragas_eval.py, switch reference metrics to ragas.aevaluate(...), and preserve current CLI commands through thin sync wrappers.

Do not try to make Phoenix async in this pass. Its experiment runner is currently used as a synchronous API, so Phoenix remains the visibility layer until its client exposes an async boundary.

Agreement Map

Contrarian

Reject async washing. Standalone Ragas first, with a semaphore.

First Principles

Move async to the true LLM judge boundary while preserving the public contract.

Expansionist

Sees future value in timeouts, resumability, and nightly async runs, but reviewers defer that.

Outsider

Make the async boundary explicit and remove hidden event-loop bridges.

Executor

Add async scorers, async Ragas runner, and focused parity tests first.

Implementation Decision

  • Add async row scorers and async row evaluators in the shared Ragas helper module.
  • Await native Ragas async methods directly; offload sync-only scorers with anyio.to_thread.run_sync.
  • Use bounded concurrency and stable result reassembly, so JSONL output order remains comparable.
  • Use ragas.aevaluate(...) for reference metrics.
  • Keep sync wrappers for existing tests, Phoenix callbacks, and CLI compatibility.
  • Add Ragas API guards so missing async APIs fail clearly.
Advisor Responses

Contrarian

The fatal mistake is async washing. Make standalone Ragas async first, preserve ordering, use a semaphore, and do not fake Phoenix async.

First Principles

The real problem is slow LLM judge work pretending to be synchronous. Put async at the scorer boundary and keep the public contract stable.

Expansionist

The async path can become a reusable eval engine for smoke runs, nightly runs, and calibration gates, but those extensions should follow parity.

Outsider

Make the async boundary boring: one async core, one outer sync bridge, no hidden event-loop bridge inside helper internals.

Executor

Change ragas.py and run_ragas_eval.py, keep the CLI command the same, and test ordering, skips, aevaluate, and Phoenix shapes.

Peer Review Highlights
  • Strongest response: the explicit async-boundary plan with thread offload for sync-only scorers.
  • Biggest risk: adding timeout/resume knobs before proving parity.
  • Missed issue: artifact and trace correlation must stay stable under concurrency.
  • Missed issue: guard against installed Ragas API drift around aevaluate, ascore, and single_turn_ascore.
