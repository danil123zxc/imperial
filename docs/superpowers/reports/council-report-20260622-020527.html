Council Verdict

The current local code is already pointed in the right direction: a compact domain-first tree using Phoenix-native span kinds. The immediate problem is runtime truth. The running Docker app image is stale, and the live Phoenix project is dominated by older noisy traces, so Phoenix still mostly shows the old mixed tree.

Do not start with another broad trace refactor. Start by rebuilding the app image, generating a fresh trace, and proving the live trace matches the source tree. Then make the compact tree a maintained contract with provenance, privacy checks, and a programmatic golden-trace smoke test.

Recommended Plan

  1. Task 0: Live trace parity. Rebuild and restart only the app container. Verify `/app/src/imperial_rag/tracing.py` contains `IMPERIAL_RAG_TRACE_SUPPRESS_INTERNALS`, `IMPERIAL_RAG_TRACE_AUTO_INSTRUMENT`, and that `/app/src/imperial_rag/retrieval.py` emits `retrieval.final_evidence` instead of the old merge/fuse spans.
  2. Generate a fresh validation trace. Run one known Streamlit query with `IMPERIAL_RAG_TRACE_AUTO_INSTRUMENT=false`, `IMPERIAL_RAG_TRACE_SUPPRESS_INTERNALS=true`, candidate documents off, and full final evidence off.
  3. Use a clean comparison surface. Either validate in a fresh Phoenix project such as `imperial-rag-readable-v2`, or record a cutoff timestamp/run tag and ignore older spans. The current `imperial-rag` project contains many obsolete traces.
  4. Lock the reader-facing tree. The expected query trace should be `imperial_rag.query` -> `retrieval` -> `retrieval.vector_search`, `retrieval.keyword_search`, `retrieval.rerank`, `retrieval.final_evidence`; and `answer.generate` -> `answer.call_model`, `answer.citation_check`.
  5. Add root provenance. Attach git SHA, Docker image tag or digest, build time if available, trace schema version, Phoenix project, run ID, and effective tracing flags to the root span so stale-image problems are visible inside Phoenix.
  6. Improve payload readability, not span count. Keep the compact tree. Add concise fields for normalized query, retrieval statuses, candidate counts, final evidence count, source labels, rerank model/top-k/top scores, LLM model/tokens, answer length, and citation pass/fail.
  7. Preserve privacy defaults. Any new field must respect `OPENINFERENCE_HIDE_*` settings. Keep raw candidate documents and full final evidence behind explicit opt-in flags.
  8. Make it machine-verifiable. Add a golden trace smoke test that queries Phoenix after a known run and fails if expected spans/kinds/provenance are missing, framework noise appears under default suppression, or redaction behavior regresses.

Contrarian

The plan fails if it treats readability as only a source-code issue. Phoenix currently shows old noisy spans, and the running container is stale. Make live trace parity Task 0 before tuning attributes.

First Principles

The trace should answer: what question ran, what evidence was gathered, how it was narrowed, what model was called, and whether citations passed. The compact tree is already the right product shape.

Expansionist

Phoenix can become the explanation surface for the whole RAG system. Improve payload clarity inside the compact tree and add a golden trace regression guard.

Outsider

"Current tracing" is ambiguous: source code and live Phoenix disagree. Rebuild, validate after a cutoff, and keep old traces from shaping the verdict.
